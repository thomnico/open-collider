# Contributing to Open Collider

## Setup

```bash
git clone https://github.com/CL-ML/open-collider.git
cd open-collider
pip install -e ".[dev]"        # Skill mode only
pip install -e ".[api,dev]"    # API mode (adds Anthropic SDK)
```

## Running tests

```bash
PYTHONPATH=src python -m pytest tests/ -v
```

Always use `PYTHONPATH=src` to ensure the correct package is resolved.

## Architecture

Open Collider supports two execution modes:

- **Skill mode**: Claude Code makes LLM calls as subagents. No API key needed.
- **API mode**: Python orchestrates everything via the Anthropic SDK. Requires API key.

Both modes share the same prepare/parse layer:

- `src/open_collider/skill_interface.py`: prepare prompts + parse responses (used by both modes)
- `src/open_collider/brainstorm.py`: Python orchestrator (API mode only)
- `src/open_collider/llm/client.py`: Anthropic API client (API mode only)
- `src/open_collider/strategies/`: domain generation strategies
- `src/open_collider/phases/`: idea generation and scoring (prompt assembly + parsing)
- `src/open_collider/scoring/`: scoring data loader and per-axis score parser
- `src/open_collider/data/`: shared YAML config and prompt templates
- `.claude/commands/`: Claude Code skills (orchestrate in skill mode, launch Python in API mode)

## Guidelines

- All code and content in English. Zero French.
- Axis weights are configurable per project (in `project_config.yaml`); axis names are fixed (5 axes defined in `src/open_collider/scoring/score_parser.py`). Changing axis names requires updating the parser, the scorer, and the judge prompt template together.
- Keep base dependencies minimal (`pyyaml` only). API deps are optional (`pip install .[api]`).
- The prepare/parse layer (`skill_interface.py`, strategies, phases) must never import from `llm/` or `brainstorm.py`. This keeps skill mode working without API deps.

## Known limitations

### DeepenStrategy: `set_id` collision risk across iterations

`DeepenStrategy._extract_families()` in `src/open_collider/strategies/deepen.py` looks up the parent family of each loved idea by `set_id` against a flat `domain_history` list. Because `set_id` (e.g. `DS1`, `DS2`) is regenerated independently by each strategy at each iteration, the same string can refer to different families across the run. If a user loves an idea generated by `deepen` or `refresh` at iteration N and the same `set_id` was used by a `fresh` family at iteration M < N, the deepen pass at iteration N+1 will follow the older family rather than the actual parent.

The bug is silent: the pipeline still produces ideas, just from a different family than intended. It only surfaces when iterating ≥3 times with mixed strategies.

**Fix path** (post-launch, requires test coverage on multi-iteration scenarios):
- Either persist `(iteration, strategy)` metadata directly on each idea so the lookup is unambiguous,
- or namespace `set_id` globally (e.g. `iter02_deepen_DS1`) so the flat lookup remains correct.

Tracked separately; do not patch around it without a regression test.
